## 0. 문제 정의 (검증된 사실)
- ✅ **맵은 해결됨**: `assets/world/sokcho_map.png` = 그림책·동화풍(둥근·파스텔·수채) 일러스트, 학교·놀이터·상점·연못·바다까지 풍부. `world_img.png`로 게임 통합 확인. AI 생성(`generate-assets.mjs`) 경로로 제작됨.
- ❌ **캐릭터가 화풍이 붕 뜸**: `world_img.png` 확대 결과, 캐릭터가 **딱딱한 픽셀/스프라이트 소녀**로 부드러운 수채 맵 위에 "얹혀" 보임. 통증이 "이모지"→**"화풍 불일치"**로 이동. 이게 남은 핵심 퀄리티 격차.
- 🎯 목표: 9명 캐스트를 **sokcho_map.png와 동일한 그림책 톤**으로 통일 재설계.

### 0.1 추가 이슈 (사용자 보강, 2026-06-24)
- **NPC가 이상함**: 현재 정지 이모지/어색한 배치·근접 트리거. → 일러스트 토큰 + 자연스러운 자세·행동 필요.
- **돌아다니는 친구들 부재**: 동네가 텅 빈 느낌. → **앰비언트(배회) 주민 NPC**로 생동감(§3.5).
- **실제 정보 연계 약함**: 게임이 실제 속초와 안 묶임. → NPC 대사·카드를 **실제 속초 정보(사진·팩트·위치)**와 연결(§3.6).
- **이미지 새로 생성**: 완성된 맵 톤을 레퍼런스로 **캐릭터 이미지를 전부 새로 생성**(§4).

## 1. 핵심 난제와 정면 분석
**AI 이미지 생성은 "한 장의 일관된 일러스트"는 잘 뽑지만, "여러 프레임·여러 각도가 일관된 4방향 걷기 스프라이트 시트"는 신뢰도가 낮다.** 탑다운 이동은 보통 4방향 걷기 애니를 원하므로 정면 충돌.
→ 해법: **이동에 4방향 걷기 시트를 요구하지 않는 캐릭터 표현**을 택해, AI의 약점을 회피한다.

## 2. 디자인 옵션

### Option A — 일러스트 "토큰" + 포트레이트 (권장)
- **월드**: 캐릭터당 **¾뷰 일러스트 1장**(AI, 고정 스타일+시드). 이동 시 **상하 bob + 좌우 플립 + 부드러운 드롭섀도 + 살짝 스쿼시**로 생동감. **4방향 걷기 시트 없음.**
- **대화**: 캐릭터당 **일러스트 포트레이트(큰 얼굴) + 표정 2~3종**.
- 장점: AI의 강점(단일 일관 일러스트)만 사용 → 맵과 톤 100% 일치 쉬움. 제작 빠름. 아늑한(cozy) 모바일 게임에서 흔한 검증된 방식.
- 단점: 위/아래를 바라보는 방향 표현이 없음(좌우 플립만). 정통 RPG 걷기 느낌은 약함.

### Option B — 일러스트 4방향 정지뷰(걷기 시트 없음)
- 캐릭터당 front/back/left/right **4장**을 AI로. 이동 시 방향에 맞는 장 + bob.
- 장점: 방향감 있음. 단점: **뒷모습 일관성**이 AI에서 흔들림, 비용 4배, 9명×4장=36장 관리.

### Option C — 2프레임 홉(hop) 스프라이트
- 캐릭터당 front + step 변형 2장, 좌우 플립 + 교대. 장점: 약간의 걷기감. 단점: 여전히 위/아래 없음, 일관성 관리 필요.

### Option D — 기존 픽셀 스프라이트에 페인터리 필터
- 보유 LPC/Kenney 스프라이트에 후처리. 장점: 즉시·무료. 단점: **진짜로는 안 맞음**(픽셀 격자 잔존), 임시방편.

**권장: Option A.** 느낌의 90%는 포트레이트+표정에서, 월드 일치는 ¾뷰 토큰에서 확보. 비용 대비 톤 일치 최고이고 AI 약점을 회피.

## 3. 캐스트(9명) 디자인 명세
모두 **공통 스타일 앵커**를 공유(아래 §4). 2.5등신 내외, 둥근 실루엣, 따뜻한 파스텔.

| # | 캐릭터 | 핵심 외형(실루엣 단서) | 톤/표정 기본 | 월드 토큰 | 포트레이트 |
|---|------|------|------|---|---|
| 1 | 아빠 | 키 큼, 모자/백팩, 다정한 눈 | 든든·미소 | ✅ | ✅ +걱정/웃음 |
| 2 | 아이(플레이어) | 작은 키, 노란 옷, 큰 눈 | 호기심·씩씩 | ✅ | ✅ +놀람/기쁨 |
| 3 | 새 친구 | 또래, 머리띠, 밝은 색 | 명랑 | ✅ | ✅ +웃음 |
| 4 | 도서관 선생님 | 안경, 책, 차분한 옷 | 따뜻·조용 | ✅ | ✅ +미소 |
| 5 | 시장 아주머니 | 앞치마, 푸근한 체형 | 푸근·웃음 | ✅ | ✅ +권유 |
| 6 | 버스 기사님 | 제복·모자 | 친절 | ✅ | ✅ |
| 7 | 약사님 | 흰 가운 | 부드러움·안심 | ✅ | ✅ |

| 9 | 갈매기 | 새, 큰 눈, 유일 판타지 상징 | 귀여움·발랄 | ✅(통통 떠다님) | ✅ |

## 3.5 돌아다니는 동네 사람 (앰비언트 NPC) — 생동감
퀘스트 NPC와 별도로, **동네에 살아 움직이는 주민**을 둬서 "텅 빈 맵" 느낌을 없앤다.
- **구성(예 5종)**: 등교하는 아이, 장바구니 든 어른, 강아지 산책하는 사람, 자전거 탄 아이, 빗자루 쓰는 가게 주인.
- **행동**: 도로/산책로 웨이포인트를 따라 **느린 왕복/랜덤 워크**(기존 `game.js` tween 패턴), 가끔 멈춰 두리번. 충돌 없음(플레이어 통과 가능)·대사 없음(탭 시 짧은 플레이버 한 줄 옵션).
- **스타일**: §4 공통 앵커 공유(같은 그림책 톤). 토큰 1장씩 + 좌우 플립.
- **밀도**: 월드맵에 동시 4~6명, 구역별로 분산(학교 구역=아이들, 시장=어른). 과밀 금지(5~9세 산만 방지).
- **연동**(`session/game`): 앰비언트 스포너 + 웨이포인트. 데이터는 §5 `data/characters.js`의 `AMBIENT[]`.

## 3.6 실제 정보 연계 (실제 속초와 묶기)
캐릭터가 **실제 속초 정보를 전달**해 "진짜 동네" 감각을 만든다. (이미 보유: `assets/photos/` 실제 사진 15장, `places.js`의 `fact`/`parentTip`; 별도 세션의 OSM 실제 지도 작업 `map-osm-integration-handoff.md`.)
- **NPC 대사 = 실제 팩트**: 시장 아주머니→닭강정 골목(실제), 도서관 선생님→속초시립도서관 어린이실, 갈매기→속초 바다, 약사님→동네 약국 위치 등. 각 NPC 대사에 **검증된 사실 1줄** 포함.
- **장소 카드 = 실제 사진**: NPC 경험 종료 시 `assets/photos/`의 **실제 사진 + 출처(ⓒ) + 실제 팩트** 카드 노출(기존 ui 흐름 활용).
- **위치 정합**: 앰비언트/NPC 배치를 OSM 실제 지도 좌표(타 세션 산출)와 정합 → 학교는 학교 구역, 시장 NPC는 시장 위치.
- **데이터 출처 규약**: 사실은 `assets/external_v3/photos/credits.json`·`places.js fact`에 근거. 미검증 정보 금지(아동 대상 신뢰성).

## 3.7 인상(친근감) 시스템 — 대항해시대2式 (정착감 핵심)
참조: 대항해시대2의 항구 주민은 **루틴대로 자연스럽게 이동**하고, **대화를 거듭하면 관계/인상이 변한다.**
아이 버전으로 번안 — *"낯선 사람 → 아는 동네 사람"* 으로 인상이 바뀌는 것이 이 게임의 정서적 목표.

- **루틴 이동(§3.5 강화)**: 앰비언트 주민은 랜덤이 아니라 **하루 루틴 경로**(아침=등굣길, 낮=시장/도서관, 저녁=귀가)를 따라 이동. 시간대(또는 진행 단계)에 따라 동선 변경 → 동네가 "살아 있는 하루"로 보임.
- **인상 단계**: 각 인물 `처음(낯섦) → 인사 → 대화 → 친해짐` 4단계. 만날 때마다 한 단계씩.
  - 시각 피드백: 포트레이트 표정이 점점 따뜻해짐, 머리 위 인상 아이콘(❓→🙂→😊→💛), 재방문 시 **아이를 알아보고 인사**.
  - 보상: 인물별 **인상 카드**("처음엔 무서웠는데, 알고 보니 다정한 약사님") — 기존 일상 카드 시스템과 통합.
- **대화로 인상 바뀜**: 첫 대화는 소개, 다음 대화부터 **따뜻한 개인 사실**(좋아하는 것·동네 이야기) 공개 → 인상 상승. 실제 속초 정보(§3.6)를 인상 상승의 "발견"으로 엮음.
- **정착 메시지**: 모든 인물이 `친해짐`이 되면 "이제 이 동네에 아는 사람이 많아졌어요" → 결과 하루지도와 연결.
- **데이터**(`data/characters.js`): 인물별 `impression: {stage, lines[stage], card}`. 진행 상태는 UI 수집 상태와 함께 저장.

## 3.8 이스터에그 — "태초마을"
- 숨은 인물(예: 산책 할아버지 또는 갈매기)과 **친해짐 단계**에서, 또는 특정 표지판 탭 시:
  > "쉿… 옛날 옛적 포켓몬 GO 초창기엔, 여기가 **태초마을**이었대." 😉
- 톤: 부모·형아 세대를 위한 윙크(아이에겐 무해한 농담). 빈도 낮게(희귀 발견감), 1회.
- 배치: 메인 흐름과 분리된 보너스 — 정착 서사를 해치지 않게.

## 4. 공통 스타일 앵커 (생성 일관성의 핵심)
**핵심 사실:** 맵·사진·내부 등 기존 비주얼은 전부 `generate-assets.mjs`의 **OpenAI `gpt-image-1`** +
공유 상수 `STORYBOOK`("warm pastel storybook illustration, soft friendly colors, cozy, child-friendly, no text…")
으로 만들어졌다. 캐릭터도 **같은 모델·같은 `STORYBOOK` 상수**를 재사용해야 맵과 화풍이 보장된다.
(새 STYLE_PREFIX를 만들지 말고 기존 `STORYBOOK`을 그대로 접두로 쓴다.)

- **동일 구도**: 토큰=정면 ¾뷰 전신, 포트레이트=가슴 위 정면, 배경 투명/단색.
- **⚠️ seed 없음**: `gpt-image-1`은 seed 파라미터가 없어 9명 상호 일관성을 시드로 못 잠근다.
  대신 ① 강한 `STORYBOOK` 접두 + ② **동일 프롬프트 골격**(외형 단서만 교체) + ③ **캐릭터 레퍼런스 시트**
  (한 장에 9명을 함께 그린 뒤 잘라 토큰/포트레이트로 사용)로 일관성 확보. 약간의 편차는 감수.
- **일관성 보강 카드(필요 시)**: 위 방법으로도 톤이 흔들리면, **레퍼런스-이미지를 받는 모델**
  (예: OpenRouter 경유 Gemini image)에 `sokcho_map.png`를 레퍼런스로 물려 보강. (기본은 gpt-image-1.)
- 산출 규격: 토큰 PNG(예: 96×128, 투명), 포트레이트 PNG(예: 256×256). 정확 규격은 `game.js`/`ui.js` 슬롯과 협의.

## 5. 소유 경계 & 연동 계약 (확정: 새 모듈 + 계약)
> 본 세션 = **캐릭터 디자인·인터랙션 세션**. 충돌 회피를 위해 **신규 모듈만 소유**하고,
> 기존 game/ui는 `window.CHARACTERS` 계약만 호출한다(그들의 파일은 내가 안 고침).

### 5.1 내 소유 (신규)
| 파일 | 역할 |
|------|------|
| `characters.js` (신규 런타임 모듈) | `window.CHARACTERS` 구현 — 토큰/포트레이트 렌더 헬퍼, 앰비언트 스포너+루틴 이동, 인상 시스템 |
| `data/characters.js` (신규 데이터) | 9 캐스트 + 앰비언트 + 인상(`impression`) + 이스터에그 데이터 → `window.CHARACTERS.cast/AMBIENT` |
| `assets/characters/` (신규 네임스페이스) | 캐릭터 토큰/포트레이트/표정 PNG (data-assets와 사전 합의로 서브폴더 카브) |

### 5.2 `window.CHARACTERS` 계약 (game/ui가 소비)
| API | 제공(나) | 소비 | 설명 |
|-----|---------|------|------|
| `CHARACTERS.makeNPC(scene, place)` | characters.js | game.js | 퀘스트 NPC를 일러스트 토큰+bob/flip/shadow로 생성 |
| `CHARACTERS.spawnAmbient(scene, region)` | characters.js | game.js | 배회 주민을 루틴 경로로 스폰(§3.5/3.7) |
| `CHARACTERS.portrait(id, expr)` | characters.js | ui.js | 대화창용 일러스트 얼굴+표정 반환(없으면 이모지 폴백) |
| `CHARACTERS.impression(id)` / `.advance(id)` | characters.js | ui.js | 인상 단계 조회/진행 + 해당 단계 대사·카드 |
| `CHARACTERS.onImpressionChange(cb)` | characters.js | ui.js | 인상 변동 시 표정/아이콘 갱신 콜백 |

- **game.js**(session/game): NPC/플레이어 스프라이트 생성부에서 `CHARACTERS.makeNPC`/`spawnAmbient` **호출만 추가**(렌더 로직은 내 모듈). 4방향 걷기 애니 제거는 그쪽이 호출 교체로 처리.
- **ui.js**(session/ui): `drawAvatar()`를 `CHARACTERS.portrait(id, expr)` 호출로 교체, 인상 단계 표시.
- **index.html**(공유): `<script src="data/characters.js">`·`<script src="characters.js">`를 `places.js` 뒤·`game.js` 앞에 추가 → **SHARED-LOCKS 점유 후** 통합 세션과 조율(직접 수정 금지).

### 5.3 에셋 생성 분담 (그림=gpt-image-1, 텍스트=OpenRouter)
